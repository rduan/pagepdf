DataHub Architecture Overview

DataHub is a modern data catalog built to enable end-to-end data discovery, observability, and governance. This document provides a comprehensive overview of its architecture and key components.

Introduction

The architecture of DataHub is designed with scalability, extensibility, and maintainability in mind. It follows a microservices architecture pattern, allowing different components to scale independently based on workload requirements.

At its core, DataHub consists of three main layers: the ingestion layer, the serving layer, and the presentation layer. Each layer has distinct responsibilities and can be deployed and scaled independently.

Core Components

Metadata Service (GMS)

The Metadata Service, also known as GMS (Generalized Metadata Service), is the primary backend service that powers DataHub. It provides RESTful APIs for reading and writing metadata, and it's responsible for enforcing access controls and maintaining data consistency.

GMS interacts with several storage systems:

  • Primary Store: MySQL or PostgreSQL for storing versioned metadata
  • Search Index: Elasticsearch for full-text search capabilities
  • Graph Store: Neo4j (optional) for relationship queries
  • Message Queue: Kafka for event streaming

Frontend Service

The Frontend Service is a Node.js application that serves the React-based web interface. It acts as a proxy between the UI and the backend services, handling authentication, session management, and providing a GraphQL API for efficient data fetching.

Metadata Ingestion

DataHub's metadata ingestion framework is plugin-based and supports extracting metadata from a wide variety of data sources. The ingestion process can be run as: 

datahub ingest -c recipe.yml

Popular sources include: MySQL, PostgreSQL, MongoDB, Kafka, Airflow, dbt, Looker, Tableau, and many more.

Data Model

DataHub's data model is built on a concept called "entities" and "aspects". An entity represents a metadata object (like a dataset, dashboard, or user), while aspects represent different facets of information about that entity.

Key Entities

  • Dataset: Represents tables, views, or collections
  • Chart: Visualization artifacts
  • Dashboard: Collections of charts
  • DataJob: ETL jobs or data pipelines
  • DataFlow: Workflows containing multiple jobs
  • CorpUser: User entities
  • CorpGroup: User groups

Scalability Considerations

DataHub is designed to scale horizontally. Each component can be scaled independently:

  • GMS can be deployed as multiple replicas behind a load balancer
  • Frontend service supports horizontal scaling with session affinity
  • Elasticsearch can be clustered for increased search performance
  • Kafka topics can be partitioned for parallel processing

Security and Authentication

DataHub supports various authentication mechanisms including:

  • Native username/password authentication
  • OIDC (OpenID Connect) integration
  • LDAP integration
  • SAML 2.0 single sign-on

Authorization is handled through a policy-based system that allows fine-grained control over who can view, edit, or manage different types of metadata.

Conclusion

DataHub's architecture provides a solid foundation for building a modern data catalog. Its modular design, extensive plugin system, and support for various deployment options make it suitable for organizations of all sizes.

For more detailed information about specific components, deployment options, or development guidelines, please refer to the respective documentation sections.
